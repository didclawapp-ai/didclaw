# DidClaw 0.x 实施计划

> **当前版本**：v0.6.1  
> **目标**：在 1.0 正式发布前，以"普通用户高频需求优先"为原则，完成所有基础功能。  
> **原则**：参考 QClaw（腾讯）的设计理念——屏蔽技术细节，只暴露用户真正关心的东西。

---

## 设计参考：QClaw vs DidClaw 差距分析

QClaw 是腾讯基于同一个 OpenClaw 生态打造的消费级产品，其核心设计哲学是**零门槛、本地安全、越用越懂你**。对比截图，DidClaw 在以下方面有明确差距：

| 功能 | QClaw | DidClaw 0.6.1 |
|------|-------|--------------|
| 历史用量统计（多会话汇总） | ✅ 对话列表 + Token 总计 | ❌ 仅当前会话 |
| 内置技能管理（Skills UI） | ✅ 61 个内置技能，可开关 | ❌ 无 |
| 开机自启 | ✅ | ❌ |
| 防休眠（长任务保活） | ✅ 开关 | ❌ |
| 定时任务 UI | ✅ 侧边栏入口 | ❌ |
| 全局快捷键唤起 | 未知 | ❌ |
| 远控通道 | ✅ 6 个 | ✅ 4 个（持平） |
| 在线升级 | ✅ | ✅ v0.6.0 新增 |
| 托盘常驻 | ✅ | ✅ v0.6.1 新增 |

## 用户高频使用场景

### 办公场景（频率最高）

- **手机遥控电脑**：微信/飞书发消息 → AI 自动整理文件/写报告（已有渠道接入，可用）
- **长任务挂机**：月报自动化、批量处理文件，电脑**不能中途休眠**（当前缺防休眠）
- **定时自动汇报**：每天 8:00 自动跑任务推送结果（当前缺定时任务 UI）
- **追踪花费**："这个月消耗了多少 Token？"（当前只有当前会话，没有历史汇总）

### 生活场景（每日触发）

- **每天自动播天气 + 提醒**：设置一次，AI 每天主动推送（依赖定时任务 + weather skill）
- **随手一问**：按快捷键直接输入，不切窗口（当前只能点托盘图标）
- **开机就绪**：开电脑 → AI 自动就位（当前需手动启动）

### 商务场景（高价值）

- **合同/PDF 提取要点**：普通对话 + 文件工具，当前基本可用
- **安全顾虑**：限制 AI 只读、弹窗确认（Exec 审批已有，缺简单安全开关）

## 0.x 版本计划

### v0.7.0 — 用户体验三件套（当前目标）

> **主题**：让 DidClaw 像一个"随时在线的助手"，而不是需要手动启动的工具。

#### 0.7.1 — 开机自启

**价值**：用户最基础的期望——装了软件，开机就在，不用每次手动启动。

**技术方案**：
- 使用 `tauri-plugin-autostart`（官方插件，Windows/macOS 均支持）
- 入口：设置 → 通用设置 → 「开机自动启动」开关（参考 QClaw 通用设置页布局）
- 状态存储：`DIDCLAW_KV` `didclaw.general.autostart`

**工作量**：半天

**交付物**：
- [ ] `Cargo.toml` 添加 `tauri-plugin-autostart`
- [ ] `lib.rs` 注册插件
- [ ] Settings 通用设置页新增开关
- [ ] `didclaw.toml` 添加权限

---

#### 0.7.2 — 防休眠（长任务保活）

**价值**：AI 正在跑月报、文件整理等长任务时，电脑不能因为无操作进入休眠而中断。

**技术方案**：
- 使用 `tauri-plugin-prevent-sleep`（或调用系统 API：Windows `SetThreadExecutionState`）
- 两种模式：
  - **手动开关**：设置 → 通用设置 → 「防止电脑休眠」常驻开关
  - **自动模式（后期）**：检测到 AI 正在运行任务时自动激活，任务结束后释放
- 当前先做手动开关，和开机自启并列在通用设置页

**工作量**：半天

**交付物**：
- [ ] Rust 侧 `prevent_sleep` / `allow_sleep` 命令
- [ ] 通用设置页新增开关
- [ ] 托盘菜单可选显示当前状态

---

#### 0.7.3 — 历史用量统计页

**价值**：用户最关心的成本感知问题——"我花了多少钱？"。QClaw 把这个放在设置第二项，可见重视程度。

**QClaw 的设计参考**（从截图）：
- 顶部 4 个卡片：总会话数 / 总对话次数 / 今日消耗 Token / 今日剩余 Token
- 底部表格：对话内容摘要 | 对话时间 | Token 消耗（负数显示）

**技术方案**：
- OpenClaw 本身已有 `model-usage` skill，该 skill 可通过对话输出用量报告
- DidClaw 的 UI 只需读取 Gateway 提供的会话历史（已有 `/sessions` API）和每条消息的 token 信息，做聚合展示
- 路由：设置对话框新增「用量统计」Tab
- 数据来源：`GET /sessions` → 遍历每个 session 的 usage 字段，按日期聚合

**工作量**：2-3 天

**交付物**：
- [ ] `UsageStatsView.vue` 组件（卡片 + 表格）
- [ ] `useUsageStats` composable（从 Gateway API 聚合数据）
- [ ] 设置对话框新增「用量统计」Tab 入口
- [ ] 空状态处理（无历史数据时的引导文案）

### v0.8.0 — 智能化与便捷性

> **主题**：AI 主动干活，用户随手唤起。

#### 0.8.1 — 全局快捷键唤起

**价值**：在任何应用里按一个快捷键，DidClaw 窗口立刻弹出，输入完立刻隐藏。这是 AI 助手类产品的标配（Raycast、Copilot 都有）。

**技术方案**：
- `tauri-plugin-global-shortcut`（官方插件）
- 默认快捷键：`Ctrl + Shift + Space`（可在设置里自定义）
- 行为：若窗口隐藏 → 显示并聚焦输入框；若窗口已显示 → 隐藏
- 与托盘图标点击行为一致

**工作量**：1 天

**交付物**：
- [ ] 注册全局快捷键
- [ ] 设置 → 通用设置 → 快捷键输入框（支持自定义）
- [ ] 快捷键冲突检测与提示

#### 0.8.2 — 内置 Skills 捆绑 + 管理 UI

**价值**：用户安装即可用，不需要知道 ClawHub 是什么。对应 QClaw 的"技能管理"页。

**背景**：
- OpenClaw 源码 `skills/` 目录已有 50+ 个 skill
- 每个 skill 是一个 `SKILL.md` 文件（YAML frontmatter + Markdown 说明）
- Gateway 开机自动扫描加载 skills 目录

**首批内置 Skills（Windows 兼容，零额外依赖）**：

| Skill | 描述 | 依赖 |
|-------|------|------|
| `weather` | 查询天气预报（支持任意城市） | curl（Win10+ 内置） |
| `model-usage` | 查询历史 Token 用量 | Python（看 Gateway 是否内置） |
| `session-logs` | 管理历史会话记录 | 无 |
| `clawhub` | 浏览和安装更多技能 | 无 |
| `skill-creator` | 通过对话创建自定义技能 | 无 |

**FirstRun 时自动安装**：
- 检测 `~/.openclaw/skills/` 目录是否存在内置 skill
- 若不存在，从 DidClaw 安装包内的 `resources/skills/` 复制过去
- 避免覆盖用户已存在的同名 skill

**Skills 管理 UI（参考 QClaw 技能管理页）**：
- 路由：设置 → 技能管理
- 显示：emoji + 名称 + 描述 + 来源标签（内置/用户安装/ClawHub）
- 操作：开关切换（禁用时移动到 `skills/.disabled/`）
- 底部：「前往 ClawHub 安装更多」按钮（外链）

**工作量**：3-4 天

**交付物**：
- [ ] 安装包内捆绑 5 个基础 skills 资源文件
- [ ] FirstRun 逻辑：检测并复制 skills 到用户目录
- [ ] `SkillsManagerView.vue` 组件
- [ ] Rust 命令：`list_installed_skills` / `toggle_skill`
- [ ] 设置对话框新增「技能管理」Tab

#### 0.8.3 — 定时任务 UI

**价值**：QClaw 侧边栏第一个入口就是「定时任务」，说明这是普通用户最想要的功能之一。"每天 8:00 给我播天气"就是定时任务的典型用例。

**背景**：
- OpenClaw/Gateway 已有 schedule/cron 能力（skill 的 `triggers.schedule` 字段支持 cron 表达式）
- DidClaw 目前没有任何定时任务的 UI 入口

**技术方案**：
- 读取 Gateway 的定时任务列表（API 待确认）
- UI：侧边栏新增「定时任务」按钮（仅桌面端显示）
- 列表页：显示已有任务（名称 / Cron 表达式 / 最近执行时间 / 启用状态）
- 新建向导：自然语言输入（"每天早上8点"）→ AI 转换为 cron 表达式 → 确认创建

**工作量**：3-4 天（依赖 Gateway API 支持，需提前确认）

**交付物**：
- [ ] 确认 Gateway 定时任务 API 接口
- [ ] 侧边栏新增「定时任务」入口
- [ ] `ScheduledTasksView.vue`（列表 + 创建）
- [ ] 自然语言 → cron 转换（可借助当前连接的 AI 模型）

### v1.1.0 — 多 Agent 集群（1.x 阶段）

> **决定延至 1.0 后**：工程量大、测试复杂，需整体交付。

三件套必须同时做，拆开做都没有意义：

| 功能 | 说明 |
|------|------|
| **P1-1 常驻指令（Standing Orders）** | 图形化编辑 AGENTS.md 全局指令 |
| **P1-6 会话多窗口** | 弹出独立窗口查看/操作不同会话 |
| **消息路由（通信）** | 渠道 → 指定 Agent 的路由规则 |

**为什么三个必须一起**：
- 单独做路由：规则里没有多 Agent 可选，配置无意义
- 单独做多窗口：多个窗口看的仍是同一 Agent
- 单独做 Standing Orders：定义了指令但无法在 UI 中验证生效范围
- **合起来**才能实现：工作渠道 → 工作 Agent（独立窗口） / 个人渠道 → 个人 Agent（独立窗口）

**技术难点**（估计开发 + 测试 2-3 周）：
- Pinia 作用域隔离（多 Tauri 窗口各有独立 JS 上下文）
- Gateway 事件跨窗口分发（BroadcastChannel / Tauri IPC）
- `selectSession` 副作用限定在触发窗口内
