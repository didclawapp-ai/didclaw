# LClaw UI — 多代理（OpenClaw）产品与技术方案

> **状态**：讨论稿 — 用于对齐概念、分期与界面边界。  
> **版本规划**：**0.x 不实施**本文中的多代理产品能力（复杂度高，依赖配置面、会话模型与协议事件）；**计划在 1.x** 再纳入路线图并分阶段落地。0.x 维持以 **`main` 为主会话**的当前体验即可。  
> **依据**：OpenClaw 官方 [多智能体路由](https://docs.openclaw.ai/zh-CN/concepts/multi-agent)、[Configuration](https://docs.openclaw.ai/gateway/configuration) 等；本仓库当前网关客户端与会话实现。

---

## 1. 背景：为什么谈多代理

OpenClaw 的「多代理」不是单一换模型，而是**多套隔离的大脑**（工作区、`agentDir`、会话存储）加上**编排方式**（谁接客、谁并行干活、谁和谁开会）。

对 LClaw UI 的粗粒度价值可以概括成：**从「一个全能会话」演进为「可分流、可并行、可协作」的控制台** —— 具体能走到哪一步，取决于网关配置、协议事件与产品是否暴露高级能力。

---

## 2. OpenClaw 三类机制（必须区分）

| 机制 | 核心作用 | 用户感知 | 典型配置 / API |
|------|----------|----------|----------------|
| **多 Agent 路由** | 入站分流：哪路消息进哪个 `agentId` | 用户在和**被路由到的那一个主代理**对话 | `openclaw.json` 的 `agents.list`、`bindings`、渠道 `accountId` 等 |
| **子 Agent** | 同一主会话内派生后台子任务，可并行，结果回主会话 | 用户主要只和**主代理**说话；可选展示子任务进度 | 如 `sessions_spawn` 等（以当前网关版本协议为准） |
| **Agent-to-Agent** | 不同主会话（不同 `agentId`）之间多轮互发消息 | 高级/自动化协作场景 | 需在配置中**显式开启**并配白名单（默认常关闭） |

**易混点**：

- 「主 + 多个子并行」与「主派出多个子」是**同一机制（子 Agent）**的两种说法，不要当成两种独立能力。
- **路由**解决的是**外部入口**（渠道、账号、群、用户）；**WebChat / Control UI** 往往是用户**主动选 `sessionKey`**，与 bindings 并行存在，产品上要分清「选会话」与「渠道自动分流」。

## 3. 产品叙事（可选，对小白沟通）

用一句话：**把单一「超人」会话，变成由「项目经理（主代理）」带领的「专家团队（子代理 / 其他主代理）」** —— 仅作讲解用语；实施时仍以官方术语（`agentId`、会话键、bindings）为准。

**表述注意**（避免过度承诺）：

- 单代理也能多步工具调用；子 Agent 带来的是**会话级并行编排**，不是「单模型绝对只能串行」。
- 并行子任务的**墙钟时间**受限速、排队、是否真并行等影响，不宜写死倍数。
- A2A 与部分能力依赖**网关配置**，不是装完即用。

## 4. 当前仓库基线（与多代理相关的现状）

- 会话侧默认与 **`main`** 对齐：空列表时用 `agent:main:main` 作为兜底会话键（与网关 `resolveMainSessionKey` 行为一致）。实现见 `lclaw-ui/src/stores/session.ts`。
- **`chat.send`** 不带根级 `model`**；模型以网关 / 会话解析为准（与现有桌面端方案一致）。
- **`openclaw.json`** 读写策略以 `docs/lclaw-ui-桌面端专属-实现方案.md` 为准（白名单、备份等）；多代理后可能出现 **per-agent 的 `model` / `workspace` / `agentDir`**，与当前「只动 `agents.defaults` 部分字段」的范围可能冲突，需单独定边界。

## 5. 分期建议（讨论用）

### 阶段 A — 多主代理可感知（路由 / 会话维度）

**目标**：用户能明确「我在和哪个主代理说话」，且不错误假设全局只有 `main`。

**可能工作项**：

- 依赖 **`sessions.list`**（及后续是否补充「代理列表」类能力）展示 **多个 `agentId` 前缀**的会话；空列表兜底策略与多代理并存时是否仍默认注入 `agent:main:main` 需再定。
- 「本机」或会话创建流程：简单 **模式 / 代理选择**（映射到不同 `agentId` 或不同主会话键），避免让用户接触原始 ID（产品层命名即可）。
- 文档与提示：`agents.list` 中 **`default` 唯一性**（多个时仅第一个生效并告警）若在 UI 暴露，需防误配。

**非目标（本阶段可不做）**：子任务进度大屏、A2A 可视化。

### 阶段 B — 子 Agent 可观测（体验增强）

**目标**：主代理派出子任务时，用户能「看见团队在干活」，减少干等焦虑。

**可能工作项**：

- 基于网关推送的 **工具 / 子会话相关事件**（以 `gateway-client-protocol-notes` 与实机协议为准）做 **时间线或简化进度列表**。
- 与现有 `toolTimeline` 等模块的衔接方式评估（避免重复造轮子）。

**风险**：事件粒度若不足，需降级为「进行中 / 完成」等粗状态，或仅展示主会话工具调用。

### 阶段 C — Agent-to-Agent 与「专家市场」类叙事（可选 / 高级）

**目标**：多主会话协作、或「安装某类专家代理」的产品包装。

**前提**：网关开启 `tools.agentToAgent`（或当前版本等价项）并配置允许列表；安全与审核策略需单独评审。

**可能工作项**：配置引导、危险操作确认、与 `openclaw.json` 写入边界对齐。
